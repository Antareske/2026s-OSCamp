# StarryOS 网络增强项目概述

## 摘要

StarryOS 已经具备较完整的 Linux 兼容运行能力，网络功能也覆盖了真实应用运行所需的主要 socket 场景。随着系统从功能验证走向真实工作负载，网络模块的关注点正在从"能否连通、能否运行"转向"是否稳定、是否高效、是否可观测、是否可回归"。

该网络增强项目以 StarryOS 网络栈（`net/ax-net`）为核心对象，围绕可靠性、性能和测试方法三个方向展开。项目目标不是孤立优化某一个接口，而是建立一套可持续演进的网络工程体系：功能行为能够接近 Linux 预期，性能变化能够被稳定测量，瓶颈位置能够被解释，后续改动能够通过基线测试持续验证。

本文档基于当前 dev 分支状态更新，反映已完成的架构重构和待推进的性能优化工作。

## 项目定位

| 维度 | 目标 |
|------|------|
| 可靠性 | 减少网络数据面中的脆弱路径，使异常输入、突发流量、连接关闭和缓冲耗尽等场景不会放大为系统级问题。 |
| 兼容性 | 持续补齐 Linux socket 行为，使 TCP、UDP、接口查询、socket 选项、阻塞/非阻塞和事件通知更符合真实应用预期。 |
| 性能 | 降低收发路径中的拷贝、锁竞争和调度开销，提高吞吐、PPS 和多核扩展能力。 |
| 可观测性 | 建立面向吞吐、丢包、重传、PPS、CPU 消耗和锁竞争的观测手段，避免性能问题只能依赖猜测定位。 |
| 可回归 | 固化可重复执行的性能测试流程，使优化前后能够在相同环境和相同指标下比较。 |

该项目的核心价值可以概括为：把 StarryOS 网络能力从“功能可用”推进到“行为可靠、性能可测、瓶颈可解释、演进可验证”。

## 当前实现概况

StarryOS 网络实现（`net/ax-net` 0.7.3 + smoltcp 0.13.1）已经具备较完整的分层结构。用户态应用通过 Linux 风格的 socket 接口访问网络能力，内核侧负责把文件描述符、系统调用、socket 状态、网络协议和设备驱动连接起来。协议处理主要覆盖 TCP、UDP、raw socket、Unix domain socket 和 vsock 等类型，其中 TCP/UDP 路径已经能够支撑常见客户端、服务端和测试工具运行。

从能力角度看，当前网络栈已经具备以下基础：

| 能力类别 | 当前状态 |
|----------|----------|
| TCP 连接 | 支持建立连接、监听、接受连接、关闭连接、非阻塞收发和基础状态查询。 |
| UDP 通信 | 支持绑定、连接、收发、peer 过滤、消息边界和常见接收语义。 |
| socket 选项 | 已补齐多项真实应用常用的类型、协议、域、缓冲和 TCP 状态相关查询。 |
| 网络接口 | 支持多接口、per-interface 路由、环回快速路径、SO_BINDTODEVICE。 |
| 事件通知 | 已具备面向 poll/epoll 的 readiness 通知能力，支持 IRQ-safe 延迟通知。 |
| 设备接入 | 可通过虚拟网卡和部分物理网卡驱动接入网络设备，支撑 QEMU 与板级场景。 |

### 已完成的关键重构

dev 分支已完成若干重要架构改进：

1. **poll 模型重构**：引入专属 net-poll worker，数据面与 syscall 解耦，socket 方法通过 `request_poll()` + waker 异步推进协议栈，避免内联 poll 导致的重入和全局串行化。
2. **listen_table 优化**：从端口数组改为 HashMap 索引，降低端口分配与查重开销。
3. **并发与锁语义文档**：新增锁顺序、竞态窗口、deferred poll waker 机制的详细说明。
4. **多接口支持**：per-interface 路由表、DNS、孤儿 TCP 回收、环回快速路径、dual-net 测试用例。
5. **virtio-net 队列序列化修复**：修复多核竞态，保证 virtio 队列访问安全。

这说明 StarryOS 网络栈已经具备进一步优化的基础。当前主要问题不在于缺少最基本的网络能力，而在于这些能力在复杂负载、高并发、多核和长时间运行下仍需要更强的稳定性和性能保障。

## 瓶颈画像

当前 StarryOS 网络性能瓶颈可以归纳为五类。下文列出的问题中，部分已通过最近的重构缓解（标注为"已改善"），其余为待推进项。

### 1. 数据路径偏长

网络收发过程中，数据通常会经过驱动缓冲、协议栈缓冲、socket 缓冲和用户缓冲等多个阶段。每一层缓冲都可能带来额外拷贝、cache 压力和内存分配开销。对于大包吞吐，这会增加每字节 CPU 消耗；对于 64B 等小包场景，这会显著放大每包固定成本。

**当前状态**：一个 TCP 包从网卡到用户态仍经历 4+ 次拷贝（DMA→驱动 Vec `to_vec()`、驱动→Router rx_buffer、rx_buffer→smoltcp socket 环、socket 环→用户缓冲）。TX 方向类似，virtio 还有额外 staging 拷贝。

优化方向是减少中间搬运，推动缓冲复用和所有权传递，使数据尽量少拷贝、少分配、少跨层转换。

### 2. 全局串行化较明显（部分已改善）

**已改善**：poll 模型重构后，数据面已从 syscall 路径解耦，专属 net-poll worker 推进协议栈，避免了每个 socket 操作内联 poll 导致的全局串行化。listen_table 改为 HashMap 后，端口查找不再线性扫描。

**待推进**：`SOCKET_SET` 仍为全局大锁（`net/ax-net/src/lib.rs:117`），所有 socket 读写共享一把锁。多核并发收发时，不同连接仍会竞争同一组共享资源。多核不一定带来吞吐提升，反而可能因锁竞争、缓存行争用和调度切换导致性能下降。

优化方向是拆分全局锁和集中表结构，逐步引入按 socket、端口、队列或 CPU 分片的数据结构，减少无关流量之间的相互阻塞。

### 3. 批处理与多队列能力待增强（部分已改善）

**已改善**：poll 模型重构后，协议推进不再由每个 syscall 触发，具备了引入 NAPI 风格批处理的基础。

**待推进**：`RX_PREFETCH_TARGET = 1`（`net/ax-net/src/device/driver.rs`），每次仍只收一个包，缺少批量收割。驱动层仍为单队列（`QUEUE_ID0`），无法利用多队列/RSS 并行处理。

优化方向是引入批量收包（`RX_PREFETCH_TARGET` 调大至 32/64）、budget poll、中断合并，并在驱动侧支持多队列，把不同流按哈希分配到不同核独立处理。

### 4. 缓冲与背压机制仍需增强

固定容量缓冲在突发流量、小包压测和高带宽时延积场景下容易成为丢包点。如果应用层设置的缓冲参数不能有效影响底层收发窗口，真实应用也难以通过常规方式调优性能。

**当前状态**：`SOCKET_BUFFER_SIZE = 64`（`net/ax-net/src/consts.rs`），Router 收发环各仅 64 包。`SO_SNDBUF/SO_RCVBUF` 仍为固定 64KB，setsockopt 为 no-op。缺少接收窗口自动调整（auto-tuning）。

优化方向是扩大并参数化关键缓冲区，补齐 socket buffer 语义，增加按原因统计的丢包计数，并建立更清晰的背压机制。

### 5. 加速能力尚未充分利用

校验和 offload、TSO/GSO/GRO、多队列、vhost-net 等能力是虚拟化和高性能网络中的重要组成部分。当前 StarryOS 网络栈对这些能力的利用仍不充分，因此在 QEMU 和真实设备上都还有明显优化空间。

**当前状态**：smoltcp feature 未启用硬件 offload，整个 IP/TCP/UDP 校验和都在 CPU 上算。没有 TSO/GSO/GRO、没有 jumbo frame（MTU 固定 1500）。

优化方向是逐步打通虚拟网卡和物理网卡的数据面加速能力，并在测试中明确区分功能冒烟拓扑和正式性能拓扑。

## 优化路线

该项目的优化路线可以分为三个阶段。第一阶段的 poll 模型重构已在 dev 分支完成，当前处于阶段一收尾与阶段二启动的过渡期。

| 阶段 | 重点 | 预期收益 | 当前状态 |
|------|------|----------|---------|
| 第一阶段：稳定基线 | 补齐错误处理、丢包统计、测试流程和基础观测指标。 | 让网络问题可复现、可定位，降低数据面异常导致系统不稳定的风险。 | **约 40% 完成**：poll 模型重构完成，待推进批处理、per-socket 计数、panic 清理、缓冲扩容。 |
| 第二阶段：单核效率 | 减少拷贝、减少分配、扩大缓冲、优化轮询批处理。 | 提升 TCP/UDP 单流吞吐、小包 PPS 和单位 CPU 效率。 | **约 10% 完成**：poll 解耦完成，待推进去拷贝、buffer 池、offload、窗口调整。 |
| 第三阶段：多核扩展 | 拆分全局锁，引入分片结构、多队列和更细粒度并行处理。 | 改善多连接、多流和多核场景下的扩展曲线。 | **约 20% 完成**：poll 线程解耦、listen_table HashMap 优化完成，待推进 SOCKET_SET 锁拆分、多队列/RSS。 |

**综合完成度：约 30%**

这些阶段不是完全割裂的。可靠性和可观测性需要贯穿整个过程；性能优化也需要每一步都通过同一套基线方法验证，避免局部指标改善但整体行为退化。

## 性能测试体系

该项目建立的性能测试方法强调四个原则：同环境、可重复、多指标、可归因。

| 原则 | 含义 |
|------|------|
| 同环境 | 优化前后使用相同拓扑、相同 CPU 配置、相同网络参数和相同测试工具。 |
| 可重复 | 每个测试项执行 warmup 和多轮正式测量，结果使用均值和标准差描述。 |
| 多指标 | 不只记录吞吐，还记录 PPS、UDP loss、TCP retransmits、多核扩展和 CPU 相关指标。 |
| 可归因 | 性能变化需要能关联到拷贝次数、锁竞争、缓冲容量、poll 次数或设备路径变化。 |

当前性能测试主要覆盖以下负载：

| 测试类别 | 关注点 |
|----------|--------|
| TCP 单流上行 | 衡量单连接发送路径吞吐和基础协议效率。 |
| TCP 多流上行 | 观察并发连接下的锁竞争和调度开销。 |
| TCP 单流下行 | 衡量接收路径吞吐和用户态读取效率。 |
| UDP 大包 | 观察无连接大包吞吐、丢包和发送效率。 |
| UDP 小包 | 放大每包固定开销，用于暴露 PPS、锁和调度瓶颈。 |

测试结果不以单次运行的峰值作为结论，而以多轮测量后的统计结果作为依据。若某个指标标准差过高，则该数据点应被视为噪声较大，需要重新检查测试环境或扩大采样次数。

## 测试拓扑分层

网络测试拓扑会直接影响结果可信度，因此项目将测试场景分为三层。

| 层级 | 用途 | 结论可信度 |
|------|------|------------|
| 功能冒烟 | 快速确认网络连通、协议可用和测试工具可运行。 | 只用于功能确认，不用于正式性能结论。 |
| QEMU 基线 | 在固定虚拟化拓扑下进行优化前后对比。 | 可用于趋势判断和阶段性性能评估。 |
| 对照基线 | 在相同拓扑下引入 Linux 作为参照对象。 | 用于判断 StarryOS 与成熟网络栈之间的差距。 |

这种分层可以避免把受限拓扑下的数字误读为系统真实性能上限。对于 StarryOS 网络增强项目而言，短期重点是形成稳定的 QEMU 基线和同拓扑 Linux 对照；长期目标是进一步扩展到更接近真实硬件的数据面验证。

## 评估方式

每次网络优化应尽量形成固定格式的对比结果：

| 评估项 | 说明 |
|--------|------|
| 优化前基线 | 记录优化前同一测试矩阵下的吞吐、PPS、loss、retransmits 和方差。 |
| 优化后结果 | 在不改变测试环境的前提下重复同一组测试。 |
| 变化幅度 | 给出每个指标的相对提升或下降，而不是只列绝对值。 |
| 归因说明 | 解释变化来自减少拷贝、降低锁竞争、扩大缓冲、改善 poll，还是设备路径变化。 |
| 风险记录 | 标注是否存在方差过大、拓扑受限、CPU 争用或尚未覆盖的场景。 |

通过这种方式，网络增强项目能够持续积累可比较的数据，而不是只保留零散的手工测试结果。

## 预期成果

该项目最终希望形成四类成果：

- 更可靠的 StarryOS 网络实现，能够在真实应用和压力场景下保持稳定行为。
- 更高效的数据面路径，减少不必要拷贝、锁竞争、分配和轮询开销。
- 更清晰的性能基线，使每次优化都可以用统一指标客观比较。
- 更强的可诊断能力，使吞吐下降、丢包、重传和多核不扩展等问题能够被快速定位。

## 后续推进计划

基于当前 30% 完成度，建议按以下优先级推进：

### 短期（1-2 周）
1. 清理数据面 panic 路径（`net/ax-net/src/router.rs:224,245`、`service.rs:451` 等）
2. 实现 RX 批处理（`RX_PREFETCH_TARGET` 改为 32/64）
3. 添加 per-socket/device 计数器（rx/tx/drop/retrans）

### 中期（2-4 周）
4. 去拷贝优化（driver buffer 所有权传递，去掉 `to_vec()`）
5. virtio TX 去 staging 拷贝
6. 打通校验和 offload（virtio-net `VIRTIO_NET_F_CSUM/GSO`）

### 长期（1-2 月）
7. 拆分 `SOCKET_SET` 全局锁为 per-socket 或分片锁
8. 驱动多队列 + RSS
9. 搭建 QEMU+vhost 测试环境并建立 Linux 基线

详细瓶颈分析、测试方法论和 QEMU 测试方案见同目录配套文档。
