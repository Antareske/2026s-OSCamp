# fix(starry-kernel): align file sync syscalls with Linux semantics

## 背景

本次工作已经围绕 StarryOS 文件同步相关 syscall 完成了一轮 Linux 语义对齐，覆盖 `fsync`、`fdatasync`、`sync`、`syncfs` 和 `sync_file_range` 五个 syscall。

## 测例

已在 `test-suit/starryos/normal/qemu-smp1/syscall` 下新增 5 个 C 测例：`test-fsync`、`test-fdatasync`、`test-sync`、`test-syncfs`、`test-sync-file-range`。测例目录结构保持与项目现有风格一致，每个测例独立携带自己的 `CMakeLists.txt` 和 `test_framework.h`，并包含必要的中文注释说明测试节点。

这 5 个测例已经加入以下 grouped syscall 配置，纳入 StarryOS QEMU CI：

- `test-suit/starryos/normal/qemu-smp1/syscall/qemu-riscv64.toml`
- `test-suit/starryos/normal/qemu-smp1/syscall/qemu-aarch64.toml`
- `test-suit/starryos/normal/qemu-smp1/syscall/qemu-x86_64.toml`
- `test-suit/starryos/normal/qemu-smp1/syscall/qemu-loongarch64.toml`

## Linux 参考结果

新增测例已经先在宿主 Linux 上分别使用 `gcc` 和 `musl-gcc` 构建运行通过，确认测例本身与 Linux 当前行为一致，而不是针对 StarryOS 定制的期望。

宿主 Linux 基线结果表明：

- `fsync` 和 `fdatasync` 对 regular file、只读 file 和 directory fd 返回成功，对 pipe 返回 `EINVAL`
- `sync()` 调用后程序继续执行
- `syncfs` 对 file、directory、pipe 这三类合法已打开 fd 都返回 `0`
- `sync_file_range` 会拒绝非法 `flags`，并对 pipe 返回 `ESPIPE`

## StarryOS 结果

在此基础上，已经对 StarryOS 的 4 个 QEMU 架构执行 `syscall` grouped case 验证。

修复前，新增测例稳定暴露出两个 Linux 兼容性问题：

- `sync_file_range` 忽略非法 `flags`，并把 pipe fd 的错误码返回成 `EINVAL`
- `syncfs` 只接受 regular file fd，对 directory fd 返回 `EISDIR`，对 pipe fd 返回 `EINVAL`

完成内核修复后，已经重新执行：

```bash
cargo xtask starry test qemu --target x86_64-unknown-none -c syscall
cargo xtask starry test qemu --target aarch64-unknown-none-softfloat -c syscall
cargo xtask starry test qemu --target riscv64gc-unknown-none-elf -c syscall
cargo xtask starry test qemu --target loongarch64-unknown-none-softfloat -c syscall
```

4 个架构结果均通过。

## 内核修改

已修改以下内核文件：

- `os/StarryOS/kernel/src/syscall/fs/io.rs`
- `os/StarryOS/kernel/src/syscall/fs/ctl.rs`

其中，`sys_sync_file_range` 已调整为先校验 fd 是否有效，再检查 `flags` 是否只使用低 3 位，最后对非 `File`/`Directory` fd 返回 `ESPIPE`，同时继续保持当前 no-op 语义。`sys_syncfs` 已统一基于 `get_file_like(fd)` 做合法性校验，对 `File` 和 `Directory` 保留真实的 `filesystem().flush()`，对其他合法 `FileLike` 按 Linux 基线返回 `0`。
